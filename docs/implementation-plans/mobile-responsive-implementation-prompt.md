# 実装Agent用Prompt：Frontendスマホレスポンシブ対応

以下をそのまま実装Agentへの依頼文として使用する。

---

あなたはこのリポジトリのフロントエンド実装担当Agentです。作業ルートは `workspace` です。

## 最重要ルール

1. 最初に `AGENTS.md` を読み、**Feature / Scenario first** を厳守してください。
2. 一次仕様は以下です。画面・実装の判断では、必ずこの順で確認してください。
   - `docs/scenarios/student-learning-happy-path.md`
   - `docs/features/10-auto-pause-resume.md`
   - 関連するキャリブレーション、自動停止、ダッシュボードのFeature / Scenario
   - `docs/frontend/spec.md`
3. 本タスクはスマホ表示へのレスポンシブ対応です。API、認証、SignalR、MediaStream、フレーム送信、再生イベント、キャリブレーション、自動停止／再開の振る舞いを変更してはいけません。
4. PC表示を崩してはいけません。既存のPC向けレイアウトは `md` 以上で維持し、スマホ向けの変更は原則として `md` 未満に限定してください。
5. shadcn/uiの見た目を変更しないでください。利用箇所で許可される変更は配置・大きさ・overflow・既存の標準 `size` / `variant` だけです。色、文字、border、角丸、影、透明度、独自の状態スタイル、shadcn/uiコンポーネント内部の変更は禁止です。
6. 関連しないリファクタリング、依存関係の追加、既存ユーザー変更の上書き・削除をしないでください。

## 目的

受講者・管理者・開発検証の全フロントエンド画面を、最小幅320pxを含むスマホ表示で利用可能にする。PC向けの構成と既存シナリオの状態遷移を保ったまま、狭幅・低い画面・タッチ操作に対応する。

## 確定した受講ページ契約

対象: `src/frontend/app/student/session/student-session-page.tsx`

### 動画表示

- スマホ縦向きでは、動画のアスペクト比を維持して画面中央に配置する。
- 1920×1080などの16:9動画では、動画幅を画面幅に合わせる。画面上下に余る領域は黒で表示し、動画を切り取らない。
- 実装上は、`md` 未満で `object-contain` と黒背景を使用する。`md` 以上は現在のPC向け `object-cover` の表示方針を維持する。
- 動画、Header、Footer、モーダルがモバイルブラウザのURLバーにより切れないよう、動的ビューポート高を使う。固定Header/Footerは安全領域も考慮する。

### Header / Footer

- マウス、タッチ、キーボード操作があれば表示し、3秒無操作で150msのfade-out、操作時は150msのfade-inとする既存仕様を必ず維持する。
- タッチ端末でのタップも無操作タイマーをリセットする。
- 狭幅では動画名、Badge群、再生ボタン、スライダー、再生時間が切れないよう折り返し・幅調整を行う。ただしPCの現在の配置を変えない。

### Badgeの詳細表示

- PCでは、現在どおりBadge群のホバーまたはキーボードフォーカス中に直近のscoreとlevelを表示する。
- タッチ端末では、Badge群をタップすると同じ詳細を表示／非表示に切り替える。
- タップ操作はHeader/Footerの無操作タイマーをリセットする。
- 状態を示す適切なARIA属性を付け、キーボード操作を壊さない。
- 詳細表示だけで自動停止・再開、動画再生、解析結果の状態を変更してはいけない。

## 共通の実装要件

- 画面全体に意図しない横スクロールを発生させない。横幅が必要なTable、SVGグラフ、タイムラインだけは専用コンテナで横スクロールを許可する。
- `p-6` 固定の画面余白は、必要に応じてスマホで `p-4`、`md`以上で `p-6` のように調整する。
- Dialog / Alertなどの重要な操作は、320px幅・横向きの低い画面でも画面外に切れないようにする。内容が高すぎる場合は内容領域を縦スクロール可能にする。
- 現行の読み込み、認証・認可、通信エラー、空状態、disabled状態を維持する。
- 新規のUIライブラリ、CSSフレームワーク、依存パッケージは追加しない。

## 受け入れ条件

### 表示幅と向き

以下をブラウザのデバイスエミュレーションまたはE2Eで確認する。

- 320px幅の縦向き
- 360px幅の縦向き
- 390px幅の縦向き
- 844×390程度のスマホ横向き
- 768px幅
- 1024px以上のPC幅

### 共通

- 320px幅でページ全体の意図しない横スクロールがない。
- 重要な入力・ボタン・エラー・ダイアログが画面外に隠れない。
- `md`以上のPCレイアウトが、変更前の構成を維持する。
- API request、認証Cookie、CSRF、SignalR、MediaStream、フレーム送信の契約に変更がない。

### 受講シナリオ

- 16:9動画をスマホ縦向きで表示したとき、動画が画面中央・画面幅に収まり、上下に黒い余白が表示され、映像が切り取られない。
- Header/Footerはタッチ・マウス・キーボード操作で表示し、3秒無操作後に150msで非表示になる。
- タッチ端末でBadge群をタップするとscore / levelの詳細を表示／非表示にできる。
- 通常受講、キャリブレーション再試行、眠気による自動停止・再開、顔未検出Popupの状態遷移および文言は変わらない。

### 管理者画面

- ダッシュボードのフィルタ、一覧の詳細表示・削除、削除確認をスマホ幅で操作できる。
- セッション一覧の全列、詳細画面の概要、チャートの全系列・タイムライン情報が失われない。
- スコアグラフの詳細情報をタッチ端末で確認できる。

## 検証

まず対象コードのlintを実行し、次にbuildを実行する。

```sh
pnpm lint
pnpm build
```

Playwrightの既存設定とテストを確認し、可能なら受講者画面のモバイルviewportを含むテストを追加または更新して実行する。

```sh
pnpm test:e2e
```

テストやビルドが環境依存サービスの不足で実行できない場合、実行したコマンド、結果、阻害要因を明記する。`git --no-pager diff --check` は実行しない。

## 完了時の報告形式

1. 対応したページと、対応するFeature / Scenario。
2. 変更したファイルと責務。
3. スマホとPCで確認した表示幅・向き。
4. 実行したlint、build、E2Eの結果。実行できなかった場合は理由。
5. 既知の残課題または仕様確認が必要な点。

実装、必要なテスト、仕様との整合確認が終わるまで作業を継続してください。
