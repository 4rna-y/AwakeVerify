# 内容の前提

対象システムは、Webカメラ映像から受講者の覚醒状態を解析するWebシステムです。
ブラウザが640×480、5fps相当の独立JPEGフレームを送信し、BackendがBlob Storageへの保存とService Busへの投入を行います。
Python WorkerがMediaPipe Face Landmarkerで顔ランドマークを推定し、EAR、Pitch、Yaw、PERCLOSから眠気スコアを算出します。
判定結果はBackendがPostgreSQLとTransactional Outboxへ保存し、SignalRを通じて受講画面へ通知します。
危険状態では動画を自動停止し、スコアと再生イベントを教員向けダッシュボードで確認できます。

主な技術は次のとおりです。

- Frontend: Next.js、TypeScript、ブラウザMediaStream API
- Backend: ASP.NET Core、HTTPS binary frame API
- Worker: Python、MediaPipe、OpenCV
- 実行基盤: Azure Container Apps
- 非同期処理: Azure Service Bus Session
- フレーム保存: Azure Blob Storage
- 状態共有: Azure Managed Redis
- 永続化: Azure Database for PostgreSQL、Transactional Outbox
- リアルタイム通知: Azure SignalR Service

## 05 眠気推定ロジック

タイトル:
眠気推定ロジック

主メッセージ:
個人別EARと15秒の時系列状態から眠気を判定する

中央の処理フロー:

MediaPipe Face Landmarker
→ EAR / Pitch / Yaw
→ PERCLOS
→ 眠気スコア
→ Normal / Caution / Warning / Danger

表示する技術要素:

- 個人別キャリブレーション
  `EAR_threshold = EAR_open × 0.75`

- PERCLOS
  15秒スライディングウィンドウ / 最大75フレーム

- 姿勢補正
  Yawで信頼度を減衰し、Pitchでスコアを増幅

- 自動停止
  `score ≥ 0.75`

スコア式を載せる場合は、以下を正確に使用する:

w_yaw = max(1.0 − |Yaw_deg| / 45.0, 0.0)
EAR_score = min(PERCLOS / 0.5, 1.0) × w_yaw
score = min(EAR_score × (1 + 0.3 × min(Pitch_deg / 30.0, 1.0)), 1.0)

デザイン:
- 数式は1つの十分に広い領域にまとめる
- 数式を小さなカードへ押し込まない
- 数式が窮屈になる場合は、処理フローと数式を上下に分ける
- 状態レベルを派手な多色表示にしない
- 顔またはカメラのアイコンは1つだけ使用してよい

## 06 状態管理と信頼性

タイトル:
状態管理と信頼性

主メッセージ:
Workerが移動・再起動しても、Sessionの順序と状態を維持する

4つの設計要素:

1. Service Bus Session
   同一Sessionは順序処理、異なるSessionは並列処理

2. Redis + Lua
   PERCLOS更新と重複除外を原子的に実行

3. Transactional Outbox
   スコアと通知要求を同一トランザクションで保存

4. 障害引継ぎ
   lock消失・scale-in時は再配送で別Workerへ引き継ぐ

小さく追加してよい要点:

- 通知はat-least-once
- 冪等キーで重複を安全に処理
- `/health/live` と `/health/ready` を分離

ただし、追加要点によって文字量が増える場合は、4つの設計要素を優先し、小さい要点は削除すること。

デザイン:
- 4分割または2×2で整理する
- 各領域は見出しと1行説明だけにする
- 一般的な盾、鍵、チェックマークの装飾アイコンは使わない
- Service Bus、Redis、PostgreSQLなど実在する技術のアイコンだけを使ってよい

## 07 Azure負荷試験

タイトル:
Azure負荷試験

主メッセージ:
30 Session・各5fpsで、p95平均831 ms / p99平均1,054 msを記録

試験条件:

- 30 Session
- 60秒
- 5fps / Session
- 15秒 ramp-up
- Worker 12〜15 replica
- 3 slot / replica、最低36 slot

SLO:

- frame-to-result p95 ≤ 2,000 ms
- frame-to-result p99 ≤ 5,000 ms
- timeout = 0

実測結果:

- p95: 772〜873 ms、平均831 ms
- p99: 938〜1,158 ms、平均1,054 ms
- timeout: 0
- Session誤配送: 0
- retryable / permanent rejection: 0
- DLQ: 0
- 試験終了時のService Bus Active queue: 0
- 同一条件で3回実施し、すべてSLOを達成

デザイン:
- `p95 平均831 ms` と `p99 平均1,054 ms` を最も大きく表示する
- `30 Session`、`5fps`、`60秒` を試験条件として次に大きく表示する
- p95とp99は、SLO値と実測平均を直接並べて比較する
- `timeout 0`、`Session誤配送 0`、`rejection 0`、`DLQ 0` を簡潔に横並びで表示する
- 同一条件の3試験すべてでSLOを達成したことは、再現性を示す補足として小さく表示する
- 比較バーを使用する場合は、バーへ直接 `実測` と `SLO` の数値を書き、凡例を作らない
- 3回分の詳細表は作らず、範囲と平均値だけを表示する
- 成功を表すトロフィー、紙吹雪、チェックマークなどは使わない
- 数値を誇張するための装飾は行わない
