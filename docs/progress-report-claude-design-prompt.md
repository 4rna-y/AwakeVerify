# 目的
「オンデマンドビデオ教材 受講完了検証システム」の開発進捗を、既存企画スライドと同じデザインで簡潔に報告する。
企画内容そのものの再説明ではなく、現在までに進めた作業と、今後の開発につながる整理状況を伝える資料にする。

# 最重要デザイン制約
- `presentation-slide.pdf` の見た目を厳守すること。
- 白背景・黒文字を基本にすること。
- アクセントカラーは青のみ。既存スライドの青を参照し、近似値として `#1E6EEB` を使用すること。
- 無駄な装飾は避けること。
- グラデーション、過度な影、写真背景、装飾的なアイコン、過剰な文字装飾は使わないこと。
- 文字の強調は、太字と青文字だけに限定すること。
- 既存スライドと同じく、余白を広く取り、情報量を詰め込みすぎないこと。
- スライド左上に青い縦バー + タイトル、右上にページ番号を置く既存フォーマットを踏襲すること。
- フォントは既存スライドに近い太めのゴシック体。日本語は可読性優先。
- 本文は短く、1スライド1メッセージにすること。

# 内容の前提
対象システムは `proposal.md` に記載のとおり、Webカメラ映像から受講者の覚醒状態をリアルタイム監視し、眠気危険レベル時に動画を自動停止し、眠気スコアと停止イベントを教員・管理者ダッシュボードへ配信するシステムである。

# 今回強調したい進捗
以下の3点を中心に構成すること。

1. Azure開発環境を整備した
   - Devcontainerを用意した。
   - マルチコンテナ構成でAzureローカルサービス環境を整えた。
   - ローカルで、Frontend / Backend / Worker と周辺サービスを接続しながら開発できる土台を作った。
   - ローカルサービス例: PostgreSQL、Redis、Azurite、Service Bus Emulator。
   - 開発対象スタック例: Next.js、ASP.NET Core、Python Worker。

2. 仕様をスタック別に整理し、Features / Scenariosへ再構成した
   - `docs/frontend/spec.md`、`docs/backend/spec.md`、`docs/worker/spec.md` として各スタックの責務を整理した。
   - その後、実装単位だけでなくユーザー価値単位で、`docs/features/` と `docs/scenarios/` に再構成した。
   - Feature例: セッション開始、Webカメラ取得、フレーム送信、Blob保存とService Bus投入、顔認識、キャリブレーション、眠気スコア算出、リアルタイム通知、自動停止、ダッシュボード表示。
   - Scenario例: 受講者の通常受講、キャリブレーション失敗と再試行、眠気検知による自動停止と再開、顔未検出警告、教員による受講状況確認。
   - 「Frontend / Backend / Worker を作る」ではなく、「Scenarioを満たすために各スタックを実装する」開発方針に整理したことを伝える。

3. Frontend / Backend / Workerに分けて実装を途中まで進めた
   - Frontend: Next.js / TypeScriptで受講者画面の土台を作成し、学籍番号入力、動画再生画面、キャリブレーション導線、shadcn/uiベースのUI整備を進めている。
   - Backend: ASP.NET Coreでセッション開始API、WebSocketによるフレーム受信、Blob保存、Service Bus投入、セッション管理の土台を進めている。
   - Worker: PythonでService Busからの処理、フレームデコード、推論処理へつなぐ土台を進めている。
   - ただし、完成済みと断定せず「途中まで実装」「土台を構築」「接続部分を順次実装中」という表現にする。
